XFileLoader Rendering Pipeline
3D File Tools » XFileLoader Rendering Pipeline

Overview

This page documents the technical rendering pipeline used by 3D File Tools to correctly display .x files in the browser. The pipeline addresses four fundamental problems that arise when rendering DirectX .x files with Three.js.

Architecture

index.html          → Entry point, splash, theme, canvas
js/app.js           → Viewer init, model loading, z-fighting fixes
js/lib/XFileLoader.js → .x parser and geometry builder
js/lib/three.min.js → Three.js r152 runtime
js/lib/OrbitControls.js → Camera orbit/pan/zoom
js/hxxLoader.js     → Hamilton .hxx binary container decoder

Loading Flow

  1. initViewer() creates the Three.js scene, camera, renderer (with logarithmic depth buffer), lights, grid, and orbit controls
  2. loadXFile(url, ...) invokes THREE.XFileLoader to parse the .x file
  3. XFileLoader parses Frame hierarchy, vertices, faces, normals, materials, and animations
  4. Each Frame’s FrameTransformMatrix is accumulated and baked into the mesh via mesh.applyMatrix4(worldBaseMx)
  5. app.js receives the array of object.models (Three.js Mesh objects) and applies post-processing

Fix 1: Ear-Clipping Triangulation

Problem

The original XFileLoader used fan triangulation to convert n-gon faces (polygons with 5+ vertices) into triangles. Fan triangulation fans all triangles from vertex 0. This works correctly only for convex polygons. .x files commonly contain concave polygons — especially on complex mechanical parts with cutouts and notches — which fan triangulation renders incorrectly, producing:

  • Missing edges on side walls
  • Asymmetric rendering between mirrored sides
  • Spurious triangles crossing through the interior of faces

Solution

Fan triangulation was replaced with a ear-clipping algorithm implemented in triangulateFaceLocal() in js/lib/XFileLoader.js.

How Ear-Clipping Works

  1. Project to 2D — The polygon’s 3D vertices are projected onto the dominant 2D plane by computing the polygon normal (Newell’s method) and dropping the axis most aligned with it
  2. Ensure CCW winding — The signed area of the 2D polygon is computed. If it is negative (clockwise), the vertex order is reversed and a reverse-map maintains correct original indices
  3. Clip ears iteratively — An “ear” is a triangle formed by three consecutive vertices where the middle vertex is convex and no other polygon vertex falls inside the triangle. Ears are emitted and removed until only 3 vertices remain
  4. Fallback — If no ear is found (degenerate geometry), falls back to fan triangulation for the remaining vertices
  5. Fast path — Triangles (3 vertices) and quads (4 vertices) skip ear-clipping entirely

Normal synchronization: The same local triangulation order computed for vertex faces is reused for normal faces, ensuring normals stay correctly aligned with their triangles.

Fix 2: Double-Sided Materials

Problem

Three.js defaults to THREE.FrontSide rendering. Faces are only visible from one side based on their winding order. When viewing a model from behind, entire surfaces become invisible.

Solution

Every material created by the XFileLoader has side = THREE.DoubleSide, ensuring all geometry is visible regardless of camera angle or face winding direction.

var mpMat = new THREE.MeshPhongMaterial();
mpMat.side = THREE.DoubleSide;

Fix 3: Overlapping Mesh Separation (Z-Fighting Prevention)

Problem

.x files often contain multiple meshes with coplanar surfaces (e.g. a bottom plate sharing the same plane as the main body base). When two surfaces share the same depth, the GPU cannot determine which is in front, producing z-fighting: a flickering/shimmering artifact.

Solution

A multi-layered approach is used:

3a. Logarithmic Depth Buffer

renderer = new THREE.WebGLRenderer({
    logarithmicDepthBuffer: true
});

Provides much higher depth precision across the full range.

3b. Overlapping Mesh Nudge (separateOverlappingMeshes() in app.js)

For every pair of meshes, this function:

  1. Computes world-space bounding boxes
  2. Checks for intersection
  3. Containment detection: If the smaller mesh is entirely inside the larger, it is tagged userData.isContained = true for strict depth testing (see 3c)
  4. For thin non-contained overlaps: measures the thinnest overlap axis and nudges the smaller mesh away by a sub-pixel amount
const nudge = overlapThickness * 0.35 + maxMeshSize * 0.002;
models[smallerIdx].position.add(offset);

3c. Per-Mesh Render Order, Polygon Offset, and Strict Depth Test

Each mesh receives a unique renderOrder and per-material polygonOffset values:

model.renderOrder = i;
m.polygonOffset = true;
m.polygonOffsetFactor = 1 + meshIdx * 4;
m.polygonOffsetUnits  = 4 + meshIdx * 8;

For contained meshes, the depth function is changed from LessEqualDepth to LessDepth:

if (isContained) {
    m.depthFunc = THREE.LessDepth;
}

This means a contained mesh’s fragments must be strictly closer than what is already in the depth buffer. At intersection seams, the polygon offset pushes the inner mesh slightly behind the outer surface, ensuring the outer mesh always wins at the seam.

3d. Grid Separation

newGrid.renderOrder = -1;
newGrid.material.depthWrite = false;
newGrid.position.y = -size.y / 2 - maxDim * 0.002;

The grid is rendered behind all model geometry with depth write disabled.

Why These Fixes Are Universal

  • Ear-clipping handles arbitrary concave polygons with any number of vertices — not just the test model
  • DoubleSide materials ensure visibility from all camera angles regardless of mesh winding order
  • Overlap separation dynamically detects and resolves coplanar surfaces based on geometry analysis, not hardcoded positions
  • Polygon offset scales with mesh count, working correctly for models with any number of sub-meshes

No geometry is deleted or modified — all original faces, edges, and surfaces from the .x file are preserved and rendered correctly.
